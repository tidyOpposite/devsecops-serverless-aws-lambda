# Security Policy

## 1. Огляд безпеки

Цей документ детально описує стратегію та заходи безпеки, впроваджені в проекті GIF Generator. Безпека розглядається як невід'ємна частина життєвого циклу проекту, інтегрована у процеси розробки, розгортання та експлуатації. Основна мета - забезпечення конфіденційності, цілісності та доступності інфраструктури та даних шляхом проактивного управління ризиками та впровадження багаторівневих механізмів захисту.

Ми дотримуємося принципів безпеки як коду (Security as Code), мінімізації привілеїв (Least Privilege) та глибокої оборони (Defense in Depth).

## 2. Безпека інфраструктури (Infrastructure as Code - Terraform)

Інфраструктура проекту повністю визначається та керується за допомогою Terraform. Це дозволяє автоматизувати створення та оновлення ресурсів, забезпечувати консистентність конфігурацій та інтегрувати перевірки безпеки безпосередньо у процес розробки інфраструктури.

*   **Принцип найменших привілеїв (Least Privilege):**
    *   **Lambda Execution Role (`lambda_exec_role`):** Ця роль має гранульовані дозволи, що обмежують її взаємодію лише з необхідними ресурсами та діями:
        *   `logs:CreateLogGroup`, `logs:CreateLogStream`, `logs:PutLogEvents`: Для запису логів виключно у визначену групу CloudWatch.
        *   `s3:PutObject`, `s3:GetObject`: Обмежено конкретним S3 бакетом для вихідних GIF (`output_bucket`).
        *   `sqs:SendMessage`: Обмежено можливістю надсилати повідомлення лише до визначеної черги SQS Dead Letter Queue (`lambda_dlq`).
        *   `kms:GenerateDataKey`, `kms:Decrypt`, `kms:Encrypt`, `kms:ReEncrypt*`, `kms:DescribeKey`: Дозволи застосовуються тільки до конкретного ключа KMS (`encryption_key`), який використовується для шифрування даних S3, SQS та CloudWatch Logs.
        *   `xray:PutTraceSegments`, `xray:PutTelemetryRecords`: Дозволяє надсилати дані для моніторингу продуктивності та діагностики до AWS X-Ray.
    *   **GitHub Actions OIDC Role:** Ця IAM роль, налаштована в AWS, надає тимчасові облікові дані для робочого процесу GitHub Actions. Її політика довірчих відносин (Trust Policy) налаштована на довіру лише до запитів від GitHub Actions OIDC провайдера для конкретного репозиторію та гілки (`main`). Політика дозволів (Permissions Policy) має бути ретельно налаштована, щоб дозволяти лише необхідні дії для `terraform apply`, `docker push` до ECR та взаємодії з AWS CLI для перевірок.
*   **Шифрування даних у стані спокою (Encryption at Rest):**
    *   **Централізований KMS ключ:** Використовується єдиний керований клієнтом симетричний ключ KMS (`aws_kms_key.encryption_key`) для послідовного шифрування всіх підтримуваних сервісів. Це спрощує управління ключами та політиками доступу.
    *   **Ротація ключа KMS:** Увімкнено автоматичну щорічну ротацію ключа (`enable_key_rotation = true`), що є рекомендованою практикою безпеки.
    *   **Політика ключа KMS:** Політика (`data.aws_iam_policy_document.kms_key_policy`) чітко визначає, які IAM принципали (root користувач, OIDC роль) та сервіси AWS (S3, SQS, CloudWatch Logs, ECR) мають право використовувати ключ та для яких операцій.
    *   **Застосування шифрування:** Усі бакети S3, черга SQS, репозиторій ECR та групи логів CloudWatch примусово використовують SSE-KMS з цим ключем.
*   **Мережева безпека та контроль доступу:**
    *   **API Gateway:** Надає HTTPS ендпоінт за замовчуванням, забезпечуючи шифрування при передачі. CORS налаштований на дозвіл `POST` та `OPTIONS` з будь-якого джерела (`allow_origins = ["*"]`). Це типово для публічних API, але для підвищення безпеки можна обмежити джерела до конкретних доменів фронтенду, якщо це можливо. API Gateway не використовує автентифікацію чи авторизацію запитів (`authorization_type = "NONE"), покладаючись на те, що Lambda функція обробляє лише ті запити, які вона очікує.
    *   **S3 Buckets:**
        *   Бакети для логів (`log_bucket`) та вихідних файлів (`output_bucket`) мають повністю заблокований публічний доступ через налаштування `aws_s3_bucket_public_access_block`.
        *   Бакет для фронтенду (`frontend_bucket`) налаштований для хостингу статичного веб-сайту. Він дозволяє публічне читання об'єктів (`s3:GetObject`) через політику бакету (`aws_s3_bucket_policy.frontend_bucket_policy`). Налаштування блокування публічного доступу для цього бакету (`aws_s3_bucket_public_access_block.frontend_bucket_pab`) спеціально дозволяють публічну політику (`block_public_policy = false`) та не обмежують доступ лише власником при наявності такої політики (`restrict_public_buckets = false`), але блокують публічні ACL.
    *   **Версіонування S3:** Увімкнено для всіх бакетів для захисту від випадкового видалення або перезапису даних.
*   **Безпечне управління станом Terraform:**
    *   **Віддалений бекенд:** Стан Terraform зберігається в S3 бакеті, що є більш безпечним та надійним, ніж локальне зберігання. Доступ до цього бакету має бути обмежений лише авторизованими IAM принципалами.
    *   **Шифрування стану:** Бакет стану використовує шифрування (`encrypt = true` у конфігурації `backend`).
    *   **Контроль доступу:** Для production-використання рекомендується додати DynamoDB locking або інший механізм блокування стану, якщо пайплайн може запускатися паралельно.
*   **Сканування інфраструктури як коду (IaC Scanning):**
    *   Інтеграція Trivy у CI/CD пайплайн дозволяє автоматично виявляти поширені помилки конфігурації безпеки AWS у Terraform коді перед розгортанням. Налаштовані пороги (`severity: 'HIGH,CRITICAL'`) та ігнорування (`trivy:ignore`) для керованих ризиків (наприклад, публічний доступ до фронтенд-бакету).

## 3. Безпека CI/CD (GitHub Actions)

Процес CI/CD є критично важливим компонентом, який автоматизує збірку, тестування та розгортання. Захист цього процесу є пріоритетом.

*   **Автентифікація в AWS (OIDC):**
    *   Використання OpenID Connect є ключовим для уникнення зберігання статичних AWS credentials у GitHub. Workflow отримує короткоживучі тимчасові облікові дані, обмінюючи OIDC токен на IAM роль.
*   **Багаторівневе сканування безпеки:**
    *   **SAST (Bandit):** Статичний аналіз коду Python Lambda на раннє виявлення потенційних проблем (напр., використання небезпечних функцій).
    *   **SCA (Snyk):** Аналіз сторонніх залежностей Python на наявність відомих вразливостей (CVE). Пайплайн налаштований на зупинку при виявленні вразливостей високого рівня (`severity-threshold=high`).
    *   **IaC Scanning (Trivy):** Перевірка Terraform коду на відповідність кращим практикам безпеки AWS.
    *   **Container Scanning (Snyk):** Сканування фінального Docker-образу на вразливості в базовій ОС та встановлених пакетах (включаючи FFmpeg).
*   **Управління секретами:**
    *   Усі чутливі змінні (ARN ролі AWS, токен Snyk) зберігаються як GitHub Encrypted Secrets, недоступні напряму в логах workflow.
*   **Захист гілки `main`:**
    *   Рекомендується налаштувати правила захисту гілки (Branch Protection Rules) для `main`, вимагаючи перевірки статусу (включаючи кроки безпеки CI/CD) та/або рев'ю коду перед злиттям змін.
*   **Обмеження дозволів Workflow:**
    *   Файл workflow використовує мінімально необхідні дозволи (`permissions: id-token: write, contents: read`), запобігаючи несанкціонованим діям з боку workflow.

## 4. Безпека додатку (Lambda Function)

Забезпечення безпеки коду Lambda функції є критичним для запобігання вразливостям на рівні бізнес-логіки.

*   **Валідація вхідних даних:**
    *   Функція очікує `POST` запит із Base64-кодованим тілом відеофайлу.
    *   Перевіряється наявність `Content-Type`, відповідність MIME-типу дозволеному переліку та коректність Base64-декодування.
    *   Розмір декодованого відео обмежено значенням `MAX_FILE_SIZE`, щоб зменшити ризик вичерпання пам'яті або тимчасового сховища Lambda.
    *   Файл записується лише в унікальну директорію в `/tmp`, після чого директорія видаляється у блоці `finally`.
*   **Обробка помилок:** Використання `try...except` блоків для коректної обробки помилок валідації вхідних даних, Base64-декодування, виконання зовнішніх процесів (FFmpeg) та сервісів AWS (S3). Повернення загальних повідомлень про помилки клієнту (4xx для помилок клієнта/валідації, 5xx для внутрішніх помилок) без розкриття внутрішніх деталей. Використання SQS DLQ для збереження повідомлень, які не вдалося обробити.
*   **Безпека залежностей:** Регулярне оновлення залежностей у `requirements.txt` (зокрема `boto3`) та моніторинг результатів сканування Snyk.
*   **Логування та моніторинг:** Функція генерує логи, які надсилаються до CloudWatch Logs. Налаштування AWS X-Ray (`tracing_config { mode = "Active" }`) допомагає відстежувати виконання запитів та діагностувати проблеми.
*   **Конфігурація Lambda:**
    *   Налаштування таймауту (`lambda_timeout`) та пам'яті (`lambda_memory_size`) для запобігання вичерпанню ресурсів.

## 5. Безпека контейнерів (Dockerfile)

Розгортання Lambda як контейнерного образу вимагає уваги до безпеки самого образу.

*   **Вибір базового образу:** Використання офіційного образу `public.ecr.aws/lambda/python:3.9` від AWS забезпечує сумісність та регулярні оновлення безпеки від вендора.
*   **Мінімізація компонентів (Багатоетапна збірка):** `Dockerfile` використовує багатоетапну збірку. На першому етапі (`builder`) встановлюються необхідні для завантаження та розпакування FFmpeg утиліти (`curl`, `tar`, `xz`). У фінальний образ копіюються лише бінарні файли `ffmpeg` та `ffprobe`, а також залежності Python. Це значно зменшує поверхню атаки, оскільки інструменти збірки відсутні у робочому середовищі Lambda.
*   **Сканування образу (Snyk):** Інтеграція сканування образу в CI/CD виявляє вразливості в пакетах ОС та бібліотеках Python.
*   **Управління вразливостями FFmpeg:** Використовується статично зібрана версія FFmpeg з зовнішнього джерела (`johnvansickle.com`). Оскільки ця збірка не оновлюється автоматично через системний менеджер пакетів, необхідно періодично перевіряти наявність оновлень та вразливостей на сайті джерела та вручну оновлювати `Dockerfile` для використання новішої версії. Сканування контейнера може не виявити всі вразливості у статично зібраному FFmpeg.
*   **Непривілейований користувач:** Образи AWS Lambda за замовчуванням виконують код від імені непривілейованого користувача `sbx_user1051`.
*   **Immutable Tags (ECR):** Налаштування `image_tag_mutability = "IMMUTABLE"` для ECR репозиторію запобігає перезапису існуючих тегів образів, що підвищує надійність розгортань.

## 6. Безпека даних

Захист даних протягом їх життєвого циклу є ключовим.

*   **Шифрування у стані спокою (Encryption at Rest):** Як описано в розділі 2, всі постійні дані шифруються за допомогою KMS.
*   **Шифрування при передачі (Encryption in Transit):**
    *   API Gateway надає HTTPS.
    *   Всі виклики AWS SDK до сервісів AWS використовують TLS за замовчуванням.
*   **Управління доступом:** IAM політики, політики ресурсів S3 та KMS контролюють доступ до даних.
*   **Типи даних:** Проект обробляє URL відео та генерує GIF. Важливо забезпечити, щоб у логи або DLQ не потрапляла надмірна або чутлива інформація.
*   **Зберігання даних:** Вихідні GIF зберігаються в S3 бакеті з обмеженим доступом. Термін зберігання логів у CloudWatch обмежений (`retention_in_days`). Термін зберігання даних у бакетах S3 не обмежений, розглянути можливість налаштування політик життєвого циклу (Lifecycle Policies) за потреби.

## 7. Управління вразливостями та моніторинг

Проактивний підхід до виявлення та усунення вразливостей.

*   **Автоматизоване сканування:** Інтеграція інструментів SAST (Bandit), SCA (Snyk для Python), IaC Scanning (Trivy) та Container Scanning (Snyk) у CI/CD.
*   **Моніторинг логів та сповіщення:** Налаштування CloudWatch Logs. Рекомендується налаштувати CloudWatch Alarms для сповіщення про:
    *   Підвищену кількість помилок Lambda (метрика `Errors`).
    *   Надходження повідомлень до черги Dead Letter Queue (метрика `ApproximateNumberOfMessagesVisible` для DLQ).
    *   Аномальні патерни викликів або помилок (може потребувати створення власних метрик з логів).
*   **Трасування (AWS X-Ray):** Допомагає візуалізувати потік запитів та виявляти вузькі місця або помилки у взаємодії компонентів.
*   **Регулярне оновлення:** Підтримка актуальних версій залежностей Python, базових образів Docker, Terraform провайдерів та інструментів сканування. Особливу увагу слід приділяти оновленню статичної збірки FFmpeg (див. розділ 5).
*   **Аудит безпеки:** Рекомендується періодично проводити ручний перегляд коду, конфігурацій інфраструктури та налаштувань безпеки.
